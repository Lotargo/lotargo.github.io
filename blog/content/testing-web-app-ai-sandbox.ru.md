# Браузер есть, а localhost нет: тестирование веб-приложения внутри AI-песочницы

Недавно я решил проверить не очередной фреймворк, а само рабочее окружение AI-агента. Вопрос был простой: если агент видит GitHub-репозиторий, умеет работать с файлами и имеет песочницу, сможет ли он сам поднять веб-приложение и проверить его браузером?

Ответ оказался интереснее обычного «да» или «нет». Внутри окружения действительно были Node.js, Chromium и Playwright. Локальные процессы и HTTP-серверы запускались. Но прямой переход браузера на `localhost`, `127.0.0.1` и даже `file://` блокировался политикой среды.

Главный вывод: наличие браузера ещё не означает наличие полноценного browser-to-server E2E. Но это не делает песочницу бесполезной. Нужно просто разделить проверку на уровни и использовать каждый доступный канал по назначению.

## Исходная задача

Эксперимент проводился на небольшом Link Saver:

- Node.js и Express на backend;
- обычные HTML, CSS и JavaScript на frontend;
- JSON-файл как постоянное хранилище;
- `node:test` для тестов;
- GitHub-коннектор для чтения и изменения репозитория.

Я не просил агента менять код. Задача была сугубо исследовательской: получить репозиторий в песочнице, запустить сервер и понять, доступна ли настоящая браузерная проверка.

## Первое ограничение: GitHub-коннектор не равен `git clone`

Коннектор хорошо подходит для работы с удалённым репозиторием:

- читать файлы;
- получать коммиты и diff;
- создавать и обновлять файлы;
- писать изменения обратно в ветку.

Но это не обязательно означает, что репозиторий можно одной операцией материализовать в локальную файловую систему песочницы.

Прямой `git clone` также не сработал, потому что терминал не мог разрешить `github.com`. В итоге локальную копию пришлось собирать из файлов, прочитанных через коннектор.

Это важное различие:

```text
GitHub connector access != network access from the runtime
```

Агент может иметь доступ к GitHub API через отдельный доверенный канал, но при этом не иметь обычного исходящего интернета из терминала.

## Второе ограничение: runtime доступен, сеть зависимостей — нет

В песочнице был Node.js, но не было установленного `pnpm` и готового `node_modules`. Corepack присутствовал, однако скачать пакетный менеджер и зависимости из npm registry не смог из-за сетевых ограничений.

Следовательно, backend-код разделился на две категории.

Можно было запускать модули, использующие только стандартную библиотеку Node.js:

- URL-валидацию;
- JSON-хранилище;
- работу с файлами;
- создание и удаление записей;
- проверку сохранения данных после повторной инициализации.

Нельзя было сразу запустить части, зависящие от внешних пакетов:

- Express-приложение;
- Express routes;
- HTML parser из npm.

Проблема была не в невозможности тестировать backend как таковой. Проблема была только в отсутствии заранее установленных внешних зависимостей.

## Третье ограничение: Chromium есть, но навигация запрещена

В окружении нашлись Chromium и Python Playwright. Headless-браузер запускался, JavaScript выполнялся, DOM был доступен.

Но попытки открыть следующие адреса завершались одинаково:

```text
http://127.0.0.1:<port>/
http://localhost:<port>/
http://0.0.0.0:<port>/
file:///.../index.html
```

Chromium возвращал:

```text
net::ERR_BLOCKED_BY_ADMINISTRATOR
```

Это не CORS, не проблема Express, не отсутствие сервера и не ошибка Playwright. Локальный статический сервер отвечал на запросы через `curl`, но именно browser navigation блокировалась административной политикой.

Поэтому флаги вроде `--disable-web-security` или `--no-sandbox` здесь почти наверняка не решают задачу. Они управляют другими слоями защиты и не обязаны отменять URL policy.

## Что всё равно можно проверить

После разделения возможностей картина стала намного полезнее.

| Уровень | Статус | Способ проверки |
| --- | --- | --- |
| Чистая backend-логика | Доступна | `node --test`, временные директории, mock `fetch` |
| JSON persistence | Доступна | Реальная файловая система песочницы |
| Локальный HTTP-сервер | Доступен | Node `fetch`, `curl`, API test client |
| Frontend DOM и CSS | Доступны | `page.setContent()` |
| Frontend JavaScript | Доступен | Подключение реального `app.js` в Playwright |
| Browser screenshots | Доступны | Headless Chromium |
| Прямой `page.goto(localhost)` | Заблокирован | Политика окружения |
| Установка npm-пакетов | Недоступна без кэша | Нет доступа к registry |
| Настоящий browser-to-server E2E | Недоступен напрямую | Нужен мост или внешнее CI-окружение |

Самая полезная находка — `page.setContent()`. HTML можно прочитать с диска, передать прямо в браузер, затем подключить CSS и JavaScript. После этого Playwright всё ещё умеет:

- искать элементы;
- нажимать кнопки;
- вводить текст;
- проверять состояния формы;
- читать computed styles;
- собирать ошибки консоли;
- делать скриншоты.

То есть frontend можно проверять почти полноценно, если заменить сетевую часть управляемым адаптером.

## Архитектурный обход вместо взлома политики

Самый интересный вариант — не пытаться снять административную блокировку, а построить мост между браузером и Node.js.

```text
Playwright page
    -> window.fetch adapter
    -> page.exposeFunction()
    -> Node.js fetch
    -> real Express API
    -> JSON storage
```

Браузер не открывает `localhost` и не делает прямой сетевой запрос. Вместо этого frontend вызывает переопределённый `window.fetch`, который передаёт запрос в функцию Node.js. Уже Node обращается к реальному Express-серверу и возвращает результат обратно в страницу.

Упрощённая идея выглядит так:

```js
await page.exposeFunction('__backendRequest', async ({ path, method, body }) => {
  const response = await fetch(`http://127.0.0.1:${port}${path}`, {
    method,
    body,
    headers: { 'content-type': 'application/json' }
  });

  return {
    status: response.status,
    body: await response.text()
  };
});

await page.evaluate(() => {
  window.fetch = async (input, options = {}) => {
    const url = new URL(input, 'http://link-saver.test');
    const result = await window.__backendRequest({
      path: `${url.pathname}${url.search}`,
      method: options.method || 'GET',
      body: options.body
    });

    return new Response(result.body, { status: result.status });
  };
});
```

Это не стопроцентный E2E в строгом смысле, потому что сетевой переход выполняет Node, а не Chromium. Но для проверки поведения приложения ценность очень высокая:

- используется реальный frontend-код;
- работает реальный backend;
- сохраняются реальные данные;
- Playwright взаимодействует с настоящим DOM;
- можно проверить loading, errors, delete и favourites;
- административная политика браузера не нарушается.

## Правильная лестница проверок

Вместо одной большой проверки «запусти приложение браузером» я бы использовал пять уровней.

### 1. Unit-тесты чистых модулей

Проверять отдельно:

- URL parsing;
- title extraction;
- storage;
- delete regression;
- error mapping;
- state transformations.

### 2. Backend integration

Поднять сервер на случайном порту и обращаться к нему через Node `fetch` или API-клиент тестового раннера.

```text
node:test -> Node fetch -> Express -> storage
```

### 3. Frontend browser tests с mock API

Загрузить HTML через `page.setContent()`, подключить CSS и JS, а `window.fetch` заменить предсказуемым mock.

Это хорошо проверяет UI-состояния без зависимости от backend.

### 4. Browser-to-Node bridge

Связать реальный frontend с реальным backend через `page.exposeFunction()`.

Это уже почти полноценный функциональный тест всей системы.

### 5. Настоящий E2E во внешнем окружении

Оставить GitHub Actions, WSL, Codespaces или обычной локальной машине только ту проверку, которую песочница принципиально не может выполнить:

```text
Chromium -> localhost -> Express -> persistence
```

Так ограничения песочницы не блокируют разработку. Они лишь определяют, на каком уровне выполняется конкретная проверка.

## Почему это важно для AI-assisted development

Вопрос «у агента есть браузер?» слишком грубый. Правильные вопросы выглядят иначе:

- может ли браузер запускаться;
- может ли он выполнять JavaScript;
- разрешён ли `page.goto`;
- доступен ли `localhost` из browser process;
- доступен ли `localhost` из Node process;
- можно ли устанавливать зависимости;
- есть ли локальный package cache;
- может ли коннектор материализовать репозиторий;
- может ли агент делать скриншоты и DOM assertions.

Это уже не бинарная возможность, а capability matrix.

Хороший агент не должен изображать полную проверку там, где она невозможна. Он должен:

1. определить границы среды;
2. проверить каждую гипотезу отдельно;
3. выполнить максимум доступных тестов;
4. честно записать, что осталось непроверенным;
5. вынести только недоступный слой во внешнее окружение.

## Вывод

AI-песочница может оказаться намного мощнее, чем кажется после первого `ERR_BLOCKED_BY_ADMINISTRATOR`.

Да, прямой browser-to-localhost E2E может быть запрещён. Да, GitHub-коннектор может не уметь делать локальный checkout. Да, package registry может быть недоступен.

Но внутри всё ещё можно:

- тестировать чистую backend-логику;
- проверять persistence на реальной файловой системе;
- запускать локальные HTTP-процессы;
- проверять API через Node;
- рендерить frontend в Chromium;
- выполнять JavaScript и DOM assertions;
- делать скриншоты;
- связать браузер с реальным backend через контролируемый мост.

Главная ошибка — пытаться воспринимать песочницу как обычный ноутбук с браузером. Это другое окружение с отдельными каналами доступа. Как только эти каналы разложены по слоям, ограничения перестают быть тупиком и превращаются в обычную архитектурную задачу.
