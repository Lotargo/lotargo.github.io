# Код без Codex: как далеко можно зайти в обычном ChatGPT

ChatGPT на `chatgpt.com` обычно воспринимают как универсального собеседника: обсудить архитектуру, объяснить ошибку, написать небольшой фрагмент кода. Для серьёзной работы с репозиторием принято открывать Codex, потому что там есть привычные агентские инструменты, рабочее дерево, терминал и интерфейс, близкий к IDE.

Но у этой привычки есть цена. Codex и появившийся в ChatGPT режим Work используют общий агентский пул. Обычные сообщения в Chat при этом учитываются отдельно и на платных персональных планах доступны в значительно более широком объёме. Отсюда возникает практический вопрос:

> Насколько далеко можно зайти в разработке, оставаясь в обычном ChatGPT Chat и не включая Codex или Work?

Эта статья именно об этом эксперименте. Не о Codex, не о Work и не о локальном агенте в IDE. Я использовал обычный чат ChatGPT, GitHub-коннектор, файловую песочницу и доступные инструменты выполнения.

## Три разных режима, которые легко спутать

На июль 2026 года граница выглядит так:

- **ChatGPT Chat** — обычный диалог. В нём можно подключать приложения и плагины, читать данные из внешних сервисов, выполнять разрешённые действия, работать с файлами и использовать встроенную песочницу.
- **ChatGPT Work** — отдельный режим для более длинных задач, исследований и готовых deliverable-артефактов.
- **Codex** — специализированный coding agent для написания, проверки и доставки кода через ChatGPT, IDE и CLI.

OpenAI прямо указывает, что Codex, ChatGPT Work и некоторые другие агентские продукты расходуют общий agentic usage pool. Обычные сообщения Chat считаются отдельно; для Plus и Pro они обозначены как unlimited с оговоркой про abuse guardrails.

Это не означает, что Chat полностью заменяет Codex. Это означает, что между «просто поговорить о коде» и «запустить полноценного coding agent» существует большая недооценённая область.

## Почему обычный Chat подходит не только для советов

Через подключённое GitHub-приложение ChatGPT может не просто читать присланный кусок кода. В зависимости от разрешений он способен:

- читать файлы репозитория;
- просматривать историю коммитов и diff;
- создавать и обновлять файлы;
- писать изменения в ветку;
- работать с документацией, задачами и структурой проекта;
- переносить полученные файлы в песочницу для локального анализа;
- выполнять Python и системные команды в изолированном окружении.

В результате обычный чат уже способен закрывать значительную часть bounded-задач:

- подготовку `AGENTS.md`, планов и skills;
- code review;
- точечные исправления;
- создание небольших модулей;
- написание unit-тестов;
- анализ результатов тестов;
- обновление README и технической документации;
- работу с GitHub без ручного копирования файлов.

Ограничения остаются: Chat не получает доступ к железу пользователя, не видит локальную IDE, не обязан уметь поднимать Docker и может работать в окружении без исходящего интернета. Но для большинства небольших и средних задач всё это не требуется.

## Эксперимент: Link Saver в обычном Chat

Для проверки я взял небольшой Link Saver:

- Node.js и Express на backend;
- обычные HTML, CSS и JavaScript на frontend;
- JSON-файл как постоянное хранилище;
- `node:test` для тестов;
- GitHub-коннектор для работы с репозиторием.

Цель была исследовательской. Я попросил ChatGPT в режиме Chat получить код, запустить доступные проверки и выяснить, насколько далеко можно дойти до настоящего browser-to-server E2E.

## Первое ограничение: GitHub-коннектор не равен `git clone`

GitHub-коннектор отлично работает с удалённым репозиторием, но не обязательно имеет команду «клонировать всё в файловую систему песочницы».

Прямой `git clone` также не сработал: терминал не мог разрешить `github.com`. Локальную копию пришлось собирать из файлов, прочитанных через отдельный доверенный канал коннектора.

```text
GitHub connector access != network access from the runtime
```

Это важная архитектурная деталь. Интеграция ChatGPT с GitHub и сетевой доступ контейнера — разные поверхности доступа.

## Второе ограничение: runtime есть, сети зависимостей может не быть

В песочнице был Node.js, но не было готового `pnpm` и `node_modules`. Corepack присутствовал, однако не мог скачать пакетный менеджер и зависимости из npm registry.

Backend разделился на две категории.

Можно было сразу запускать код на стандартной библиотеке Node.js:

- URL-валидацию;
- JSON-хранилище;
- работу с файлами;
- создание и удаление записей;
- проверку сохранения данных после повторной инициализации.

Нельзя было без подготовки запустить части с внешними пакетами:

- Express-приложение;
- Express routes;
- HTML parser из npm.

То есть backend-тесты в обычном Chat доступны. Конкретный предел определяется не режимом Chat, а содержимым песочницы и наличием зависимостей.

## Третье ограничение: Chromium есть, но localhost может быть закрыт

В окружении нашлись Chromium и Playwright. Headless-браузер запускался, выполнял JavaScript, видел DOM, вычислял CSS и делал скриншоты.

Но прямой переход на локальные адреса блокировался:

```text
http://127.0.0.1:<port>/
http://localhost:<port>/
http://0.0.0.0:<port>/
file:///.../index.html
```

Chromium возвращал:

```text
net::ERR_BLOCKED_BY_ADMINISTRATOR
```

При этом тот же локальный HTTP-сервер отвечал через `curl`. Значит, проблема была не в приложении и не в localhost как таковом, а в политике browser navigation.

## Что реально можно проверить в Chat

| Уровень | Статус | Способ проверки |
| --- | --- | --- |
| Чистая backend-логика | Доступна | `node --test`, временные директории, mock `fetch` |
| JSON persistence | Доступна | Реальная файловая система песочницы |
| Локальный HTTP-сервер | Доступен | Node `fetch`, `curl`, API test client |
| Frontend DOM и CSS | Доступны | `page.setContent()` |
| Frontend JavaScript | Доступен | Подключение реального `app.js` в Playwright |
| Browser screenshots | Доступны | Headless Chromium |
| Прямой `page.goto(localhost)` | Заблокирован | Политика окружения |
| Установка npm-пакетов | Зависит от кэша и сети | В эксперименте registry был недоступен |
| Настоящий browser-to-server E2E | Недоступен напрямую | Нужен мост или внешнее CI-окружение |

Самой полезной находкой оказался `page.setContent()`. HTML можно прочитать с диска и передать прямо в Chromium, после чего подключить CSS и JavaScript. Playwright всё ещё может:

- находить элементы;
- нажимать кнопки;
- вводить текст;
- проверять состояния формы;
- читать computed styles;
- собирать console errors;
- делать скриншоты.

Это уже не «ChatGPT только написал код». Это реальная браузерная проверка frontend, пусть и без прямого перехода на localhost.

## Мост между браузером и реальным backend

Административную политику не обязательно ломать. Можно заменить недоступный сетевой переход архитектурным адаптером:

```text
Playwright page
    -> window.fetch adapter
    -> page.exposeFunction()
    -> Node.js fetch
    -> real Express API
    -> JSON storage
```

Браузер вызывает переопределённый `window.fetch`. Запрос передаётся в Node.js через `page.exposeFunction()`. Уже Node обращается к реальному Express-серверу и возвращает ответ странице.

Это не стопроцентный E2E в строгом смысле, потому что HTTP-переход выполняет Node, а не Chromium. Но функционально проверяются:

- реальный frontend-код;
- реальный backend;
- persistence;
- DOM и пользовательские действия;
- loading, error, delete и favourite-сценарии.

Последний настоящий маршрут `Chromium -> localhost -> Express` можно оставить GitHub Actions, WSL, Codespaces или локальной машине.

## Когда обычного Chat достаточно

Chat хорошо подходит, когда задача ограничена и её можно разложить на проверяемые шаги:

- работа с документацией и архитектурой;
- небольшая функция или исправление;
- code review и поиск регрессии;
- unit- и API-тесты;
- статический frontend;
- GitHub-операции через connector;
- анализ файлов, логов и test output;
- создание артефактов, которые не требуют полноценной локальной инфраструктуры.

Codex разумнее включать, когда нужны:

- длинная автономная работа по большому кодовому дереву;
- привычная IDE/CLI-среда;
- параллельные agents и worktrees;
- сложные refactor и migrations;
- активная установка зависимостей;
- доступ к локальным инструментам и developer browser;
- многошаговая задача, где стоимость постоянного ручного управления выше стоимости agentic лимита.

Work полезен для длинных исследований и deliverable-задач, где агент должен самостоятельно собирать большой итоговый результат.

Главная мысль: режим выбирается не по признаку «есть код или нет», а по требуемой автономности и инфраструктуре.

## Почему у OpenAI здесь редкая комбинация

Это не утверждение, что модели других компаний хуже пишут код. Речь именно о продуктовой связке.

В одном обычном ChatGPT-диалоге можно совместить:

- сильную reasoning-модель;
- почти неисчерпаемый для обычной работы лимит сообщений на Plus/Pro;
- GitHub и другие подключённые приложения с read/write actions;
- файловую песочницу;
- выполнение кода;
- браузерный runtime и скриншоты;
- запись изменений обратно в репозиторий.

Google AI Studio предоставляет мощный playground, code execution, Google Search, URL Context, Computer Use и File Search. Но это другой сценарий: официальная документация описывает AI Studio прежде всего как среду прототипирования Gemini API, а пользовательские интеграции требуют function calling и собственного исполняющего приложения. Готового GitHub-коннектора с действиями над репозиторием внутри обычного диалога в списке встроенных инструментов нет.

Официальное приложение Gemini тоже умеет многое, но его usage limits заметнее влияют на длинные итеративные сессии. Поэтому в моём сценарии сочетание «обычный чат + connectors + sandbox + широкий chat limit» пока выглядит особенно сильным именно у OpenAI.

Это оценка продуктовой эргономики на июль 2026 года, а не вечный рейтинг моделей. Лимиты и поверхности доступа могут измениться.

## Правильная лестница проверок

Вместо команды «запусти приложение и проверь браузером» лучше использовать несколько уровней:

1. Unit-тесты чистых модулей.
2. Backend integration через Node `fetch`.
3. Frontend browser tests с mock API.
4. Browser-to-Node bridge с реальным backend.
5. Один настоящий E2E во внешнем окружении.

Хороший AI-помощник не должен притворяться, что выполнил недоступную проверку. Он должен определить границы среды, выполнить максимум доступного и честно оставить только действительно невозможный слой внешнему CI.

## Вывод

Обычный ChatGPT Chat сильно недооценён как инструмент разработки.

Да, это не Codex IDE. Он не получает доступ к пользовательскому Docker, GPU и локальному терминалу. Песочница может быть без сети, а браузеру могут запретить localhost.

Но даже при этих ограничениях Chat способен:

- работать с настоящим GitHub-репозиторием;
- писать и обновлять код;
- запускать чистые backend-тесты;
- проверять persistence;
- поднимать локальные HTTP-процессы;
- анализировать API через Node;
- выполнять frontend-код в Chromium;
- проверять DOM и CSS;
- делать скриншоты;
- фиксировать результат обратно в репозитории.

Для небольшой или средней задачи Codex часто является удобным, но не обязательным инструментом. Значительную часть разработки можно выполнить в обычном Chat, сохранив общий агентский лимит Work/Codex для тех задач, где действительно нужна автономная IDE-подобная среда.
