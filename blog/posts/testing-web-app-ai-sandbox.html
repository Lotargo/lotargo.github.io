The Browser Exists, but localhost Does Not: Testing a Web App Inside an AI Sandbox

Recently I decided to test not another framework, but the AI agent's working environment itself. If an agent can see a GitHub repository, work with files, and has a sandbox, can it start a web application and verify it in a browser on its own?

The answer was more interesting than a plain yes or no. The environment really did contain Node.js, Chromium, and Playwright. Local processes and HTTP servers could run. But direct browser navigation to localhost, 127.0.0.1, and even file:// was blocked by environment policy.

The main lesson is that having a browser does not automatically mean having full browser-to-server E2E. That does not make the sandbox useless. It only means verification should be split into layers, with each available channel used for the job it can actually perform.

The starting point

The experiment used a small Link Saver:

  • Node.js and Express on the backend;
  • plain HTML, CSS, and JavaScript on the frontend;
  • a JSON file for persistence;
  • node:test for tests;
  • a GitHub connector for reading and changing the repository.

I did not ask the agent to change the code. The task was purely exploratory: get the repository into the sandbox, start the server, and find out whether real browser verification was available.

First boundary: a GitHub connector is not git clone

The connector is useful for reading files, retrieving commits and diffs, creating and updating files, and writing changes back to a branch. That does not necessarily mean the repository can be materialized into the sandbox filesystem with one operation.

A direct git clone also failed because the terminal could not resolve github.com. The local copy therefore had to be assembled from files read through the connector.

GitHub connector access != network access from the runtime

An agent may reach the GitHub API through a separate trusted channel while the terminal itself still has no ordinary outbound internet access.

Second boundary: the runtime exists, the dependency network does not

Node.js was present in the sandbox, but pnpm and a ready node_modules directory were not. Corepack existed, yet it could not download the package manager or dependencies from the npm registry because of network restrictions.

Modules using only the Node.js standard library could still run:

  • URL validation;
  • JSON storage;
  • filesystem operations;
  • creating and deleting records;
  • checking that data survives reinitialization.

Parts depending on external packages could not be started immediately: the Express application, Express routes, and the npm HTML parser. The problem was not an inability to test backend code in general. It was only the absence of preinstalled external dependencies.

Third boundary: Chromium is installed, navigation is forbidden

The environment contained Chromium and Python Playwright. The headless browser launched, JavaScript executed, and the DOM was available.

But attempts to open the following addresses all ended the same way:

http://127.0.0.1:<port>/
http://localhost:<port>/
http://0.0.0.0:<port>/
file:///.../index.html

Chromium returned:

net::ERR_BLOCKED_BY_ADMINISTRATOR

This was not CORS, an Express failure, a missing server, or a Playwright error. A local static server answered requests through curl, but browser navigation itself was blocked by administrative policy.

Flags such as --disable-web-security or --no-sandbox are therefore unlikely to solve the problem. They control different security layers and do not necessarily override URL policy.

The most useful finding was page.setContent(). HTML can be read from disk and injected directly into the browser, then CSS and JavaScript can be attached. Playwright can still locate elements, click buttons, fill inputs, inspect form states, read computed styles, collect console errors, and take screenshots.

So the frontend can still be tested almost fully if its network boundary is replaced with a controlled adapter.

Architectural workaround instead of bypassing policy

Playwright page -> fetch adapter -> Node bridge -> Express API -> JSON storage

The browser never opens localhost and never sends the direct network request. The frontend calls an overridden window.fetch, which forwards the request into a Node.js function. Node then calls the real Express server and returns the result to the page.

await page.exposeFunction('__backendRequest', async ({ path, method, body }) => {
  const response = await fetch(`http://127.0.0.1:${port}${path}`, {
    method,
    body,
    headers: { 'content-type': 'application/json' }
  });

  return {
    status: response.status,
    body: await response.text()
  };
});

await page.evaluate(() => {
  window.fetch = async (input, options = {}) => {
    const url = new URL(input, 'http://link-saver.test');
    const result = await window.__backendRequest({
      path: `${url.pathname}${url.search}`,
      method: options.method || 'GET',
      body: options.body
    });

    return new Response(result.body, { status: result.status });
  };
});

This is not 100% E2E in the strict sense because Node, not Chromium, performs the network hop. But the functional value is high: the real frontend and backend run, real data is persisted, Playwright interacts with the real DOM, and browser policy is not violated.

A better verification ladder

1. Unit tests for pure modules

Test URL parsing, title extraction, storage, deletion regressions, error mapping, and state transformations separately.

2. Backend integration

Start the server on a random port and call it through Node fetch or the test runner's API client.

node:test -> Node fetch -> Express -> storage

3. Frontend browser tests with a mock API

Load the HTML through page.setContent(), attach CSS and JavaScript, and replace window.fetch with predictable mock responses. This is ideal for UI states without depending on the backend.

4. Browser-to-Node bridge

Connect the real frontend to the real backend through page.exposeFunction(). This becomes an almost complete functional test of the whole system.

5. Real E2E in an external environment

Leave only the check the sandbox fundamentally cannot perform to GitHub Actions, WSL, Codespaces, or a normal local machine:

Chromium -> localhost -> Express -> persistence

The sandbox restrictions no longer block development. They only determine which layer performs each check.

Why this matters for AI-assisted development

The question “does the agent have a browser?” is too coarse. Better questions are:

  • can the browser launch and execute JavaScript;
  • is page.goto allowed;
  • is localhost reachable from the browser process;
  • is localhost reachable from the Node process;
  • can dependencies be installed or restored from cache;
  • can the connector materialize a repository;
  • can the agent take screenshots and perform DOM assertions.

This is not a binary capability. It is a capability matrix.

A good agent should not pretend to have completed full verification when the environment prevents it. It should identify the boundaries, test each hypothesis separately, run the maximum available verification, record what remains unverified, and move only the unavailable layer to an external environment.

Conclusion

An AI sandbox can be much more capable than it first appears after an ERR_BLOCKED_BY_ADMINISTRATOR response.

Direct browser-to-localhost E2E may be forbidden. A GitHub connector may not provide a local checkout. The package registry may be unreachable. But the environment may still support pure backend tests, real filesystem persistence, local HTTP processes, API verification through Node, frontend rendering in Chromium, JavaScript execution, DOM assertions, screenshots, and a controlled bridge to the real backend.

The main mistake is treating the sandbox like an ordinary laptop with a browser. It is a different environment with separate access channels. Once those channels are mapped by layer, the restrictions stop being a dead end and become a normal architecture problem.

Браузер есть, а localhost нет: тестирование веб-приложения внутри AI-песочницы

Недавно я решил проверить не очередной фреймворк, а само рабочее окружение AI-агента. Если агент видит GitHub-репозиторий, умеет работать с файлами и имеет песочницу, сможет ли он сам поднять веб-приложение и проверить его браузером?

Ответ оказался интереснее обычного «да» или «нет». Внутри окружения действительно были Node.js, Chromium и Playwright. Локальные процессы и HTTP-серверы запускались. Но прямой переход браузера на localhost, 127.0.0.1 и даже file:// блокировался политикой среды.

Главный вывод: наличие браузера ещё не означает наличие полноценного browser-to-server E2E. Но это не делает песочницу бесполезной. Нужно просто разделить проверку на уровни и использовать каждый доступный канал по назначению.

Исходная задача

Эксперимент проводился на небольшом Link Saver:

  • Node.js и Express на backend;
  • обычные HTML, CSS и JavaScript на frontend;
  • JSON-файл как постоянное хранилище;
  • node:test для тестов;
  • GitHub-коннектор для чтения и изменения репозитория.

Я не просил агента менять код. Задача была сугубо исследовательской: получить репозиторий в песочнице, запустить сервер и понять, доступна ли настоящая браузерная проверка.

Первое ограничение: GitHub-коннектор не равен git clone

Коннектор хорошо подходит для чтения файлов, получения коммитов и diff, создания и обновления файлов, а также записи изменений обратно в ветку. Но это не обязательно означает, что репозиторий можно одной операцией материализовать в локальную файловую систему песочницы.

Прямой git clone также не сработал, потому что терминал не мог разрешить github.com. В итоге локальную копию пришлось собирать из файлов, прочитанных через коннектор.

GitHub connector access != network access from the runtime

Агент может иметь доступ к GitHub API через отдельный доверенный канал, но при этом не иметь обычного исходящего интернета из терминала.

Второе ограничение: runtime доступен, сеть зависимостей — нет

В песочнице был Node.js, но не было установленного pnpm и готового node_modules. Corepack присутствовал, однако скачать пакетный менеджер и зависимости из npm registry не смог из-за сетевых ограничений.

Модули, использующие только стандартную библиотеку Node.js, всё равно можно было запускать:

  • URL-валидацию;
  • JSON-хранилище;
  • работу с файлами;
  • создание и удаление записей;
  • проверку сохранения данных после повторной инициализации.

Части, зависящие от внешних пакетов, нельзя было сразу запустить: Express-приложение, Express routes и HTML parser из npm. Проблема была не в невозможности тестировать backend как таковой, а только в отсутствии заранее установленных внешних зависимостей.

Третье ограничение: Chromium есть, но навигация запрещена

В окружении нашлись Chromium и Python Playwright. Headless-браузер запускался, JavaScript выполнялся, DOM был доступен.

Но попытки открыть следующие адреса завершались одинаково:

http://127.0.0.1:<port>/
http://localhost:<port>/
http://0.0.0.0:<port>/
file:///.../index.html

Chromium возвращал:

net::ERR_BLOCKED_BY_ADMINISTRATOR

Это не CORS, не проблема Express, не отсутствие сервера и не ошибка Playwright. Локальный статический сервер отвечал на запросы через curl, но именно browser navigation блокировалась административной политикой.

Поэтому флаги вроде --disable-web-security или --no-sandbox здесь почти наверняка не решают задачу. Они управляют другими слоями защиты и не обязаны отменять URL policy.

Самая полезная находка — page.setContent(). HTML можно прочитать с диска, передать прямо в браузер, затем подключить CSS и JavaScript. После этого Playwright всё ещё умеет искать элементы, нажимать кнопки, вводить текст, проверять состояния формы, читать computed styles, собирать ошибки консоли и делать скриншоты.

То есть frontend можно проверять почти полноценно, если заменить сетевую часть управляемым адаптером.

Архитектурный обход вместо взлома политики

Playwright page -> fetch adapter -> Node bridge -> Express API -> JSON storage

Браузер не открывает localhost и не делает прямой сетевой запрос. Вместо этого frontend вызывает переопределённый window.fetch, который передаёт запрос в функцию Node.js. Уже Node обращается к реальному Express-серверу и возвращает результат обратно в страницу.

await page.exposeFunction('__backendRequest', async ({ path, method, body }) => {
  const response = await fetch(`http://127.0.0.1:${port}${path}`, {
    method,
    body,
    headers: { 'content-type': 'application/json' }
  });

  return {
    status: response.status,
    body: await response.text()
  };
});

await page.evaluate(() => {
  window.fetch = async (input, options = {}) => {
    const url = new URL(input, 'http://link-saver.test');
    const result = await window.__backendRequest({
      path: `${url.pathname}${url.search}`,
      method: options.method || 'GET',
      body: options.body
    });

    return new Response(result.body, { status: result.status });
  };
});

Это не стопроцентный E2E в строгом смысле, потому что сетевой переход выполняет Node, а не Chromium. Но для проверки поведения приложения ценность очень высокая: используется реальный frontend-код, работает реальный backend, сохраняются реальные данные, Playwright взаимодействует с настоящим DOM, а административная политика браузера не нарушается.

Правильная лестница проверок

1. Unit-тесты чистых модулей

Отдельно проверять URL parsing, title extraction, storage, delete regression, error mapping и state transformations.

2. Backend integration

Поднять сервер на случайном порту и обращаться к нему через Node fetch или API-клиент тестового раннера.

node:test -> Node fetch -> Express -> storage

3. Frontend browser tests с mock API

Загрузить HTML через page.setContent(), подключить CSS и JS, а window.fetch заменить предсказуемым mock. Это хорошо проверяет UI-состояния без зависимости от backend.

4. Browser-to-Node bridge

Связать реальный frontend с реальным backend через page.exposeFunction(). Это уже почти полноценный функциональный тест всей системы.

5. Настоящий E2E во внешнем окружении

Оставить GitHub Actions, WSL, Codespaces или обычной локальной машине только ту проверку, которую песочница принципиально не может выполнить:

Chromium -> localhost -> Express -> persistence

Так ограничения песочницы не блокируют разработку. Они лишь определяют, на каком уровне выполняется конкретная проверка.

Почему это важно для AI-assisted development

Вопрос «у агента есть браузер?» слишком грубый. Правильнее спросить:

  • может ли браузер запускаться и выполнять JavaScript;
  • разрешён ли page.goto;
  • доступен ли localhost из browser process;
  • доступен ли localhost из Node process;
  • можно ли устанавливать зависимости или восстанавливать их из кэша;
  • может ли коннектор материализовать репозиторий;
  • может ли агент делать скриншоты и DOM assertions.

Это уже не бинарная возможность, а capability matrix.

Хороший агент не должен изображать полную проверку там, где она невозможна. Он должен определить границы среды, проверить каждую гипотезу отдельно, выполнить максимум доступных тестов, честно записать, что осталось непроверенным, и вынести только недоступный слой во внешнее окружение.

Вывод

AI-песочница может оказаться намного мощнее, чем кажется после первого ERR_BLOCKED_BY_ADMINISTRATOR.

Прямой browser-to-localhost E2E может быть запрещён. GitHub-коннектор может не уметь делать локальный checkout. Package registry может быть недоступен. Но внутри всё ещё можно тестировать чистую backend-логику, проверять persistence на реальной файловой системе, запускать локальные HTTP-процессы, проверять API через Node, рендерить frontend в Chromium, выполнять JavaScript и DOM assertions, делать скриншоты и связывать браузер с реальным backend через контролируемый мост.

Главная ошибка — воспринимать песочницу как обычный ноутбук с браузером. Это другое окружение с отдельными каналами доступа. Как только эти каналы разложены по слоям, ограничения перестают быть тупиком и превращаются в обычную архитектурную задачу.
