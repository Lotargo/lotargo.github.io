Coding Without Codex: How Far Regular ChatGPT Can Go

ChatGPT is usually treated as a general assistant, while serious repository work is sent to Codex. But Codex and ChatGPT Work share an agentic usage pool, while ordinary Chat messages are accounted for separately. That raises a practical question: how much real software work can be done without launching an agentic mode at all?

This article is about a test performed in regular ChatGPT Chat, not Codex, not Work, and not a local IDE agent. The tools were a GitHub connector, the file sandbox, and the execution capabilities available inside the conversation.

Three product surfaces that are easy to confuse

  • ChatGPT Chat is the normal conversation surface. It can use connected apps and plugins, work with files, execute code in a sandbox, and perform permitted external actions.
  • ChatGPT Work is a separate experience for longer research and deliverable-oriented tasks.
  • Codex is the specialized coding agent available through ChatGPT, IDE integrations, and the CLI.

OpenAI explicitly states that Codex, ChatGPT Work, and several other agentic products draw from the same agentic usage pool. Regular Chat messages are separate; Plus and Pro list messages and interactions as unlimited, subject to abuse guardrails.

The interesting space is not “chat versus coding.” It is bounded work in Chat versus autonomous work in an agentic environment.

Why regular Chat can do more than give advice

Through a connected GitHub app, ChatGPT can read repository files, inspect commits and diffs, create and update files, write changes to a branch, and bring files into the sandbox for local analysis.

That already covers a large set of practical tasks:

  • project plans, AGENTS.md, and skills;
  • code review and regression analysis;
  • targeted fixes and small modules;
  • unit tests and test-output analysis;
  • README and technical documentation updates;
  • GitHub work without manually copying files.

Chat does not receive the user's hardware, local IDE, Docker daemon, or unrestricted network. But most small and medium tasks do not require those things.

The experiment: Link Saver in regular Chat

The test project was a small Link Saver with Node.js and Express, a plain JavaScript frontend, JSON persistence, node:test, and a GitHub connector.

The goal was exploratory: obtain the code, run every available verification step, and determine how close regular Chat could get to real browser-to-server E2E.

First boundary: a GitHub connector is not git clone

The connector worked with the remote repository, but did not expose a one-click operation that materialized the entire repository into the sandbox filesystem.

A direct git clone also failed because the terminal could not resolve github.com. The local copy had to be assembled from files read through the connector's separate trusted channel.

GitHub connector access != network access from the runtime

ChatGPT's integration with GitHub and the container's outbound network are separate access surfaces.

Second boundary: the runtime exists, but the dependency network may not

Node.js was present, but pnpm and a ready node_modules directory were not. Corepack existed, yet it could not download the package manager or dependencies from the npm registry.

Code using only the Node.js standard library still ran:

  • URL validation;
  • JSON storage;
  • filesystem operations;
  • creating and deleting records;
  • persistence checks after reinitialization.

Parts requiring Express and an npm HTML parser could not start without dependencies. The boundary was therefore set by the sandbox contents, not by the Chat surface itself.

Third boundary: Chromium exists, but localhost may be closed

Chromium and Playwright were available. The browser launched, executed JavaScript, exposed the DOM, computed CSS, and produced screenshots.

Direct navigation to local addresses was blocked:

http://127.0.0.1:<port>/
http://localhost:<port>/
http://0.0.0.0:<port>/
file:///.../index.html

Chromium returned:

net::ERR_BLOCKED_BY_ADMINISTRATOR

The same local server still answered through curl. The application was running; the block existed specifically at the browser-navigation policy layer.

What can actually be verified in Chat

The most useful finding was page.setContent(). HTML can be injected directly into Chromium, then the real CSS and JavaScript can be attached. Playwright can still locate elements, click buttons, fill inputs, inspect form states, read computed styles, collect console errors, and take screenshots.

Bridging the browser to the real backend

Playwright page -> fetch adapter -> Node bridge -> Express API -> JSON storage

The browser can call an overridden window.fetch, which crosses into Node.js through page.exposeFunction(). Node calls the real Express server and returns the response to the page.

This is not strict 100% E2E because Node performs the HTTP hop, but it functionally verifies the real frontend, real backend, persistence, DOM interactions, and loading, error, delete, and favourite flows.

When regular Chat is enough

Chat is a good fit for bounded work that can be decomposed into verifiable steps: documentation, a small feature or fix, code review, unit and API tests, static frontend work, GitHub operations, and analysis of files, logs, and test output.

Codex is the better choice for long autonomous work across a large code tree, IDE or CLI workflows, parallel agents and worktrees, complex migrations, active dependency installation, local developer tools, and tasks where constant steering costs more than the agentic allowance.

Work is useful for long research and deliverable tasks where the agent needs to assemble a substantial final result independently.

The mode should be selected by required autonomy and infrastructure, not simply by whether the task contains code.

Why OpenAI has an unusual product combination here

This is not a claim that competing models are worse at coding. The point is the product combination.

A regular ChatGPT conversation can combine a strong reasoning model, a broad message allowance, connected apps with read and write actions, a file sandbox, code execution, a browser runtime, screenshots, and repository writes.

Google AI Studio offers a powerful playground, code execution, Google Search, URL Context, Computer Use, and File Search. But Google's official documentation presents AI Studio primarily as an environment for prototyping with the Gemini API. Custom integrations rely on function calling and execution in the user's own application; an equivalent ready-made GitHub repository connector with write actions is not listed among the built-in tools.

The official Gemini app is also capable, but its usage limits are more visible during long iterative sessions. In this workflow, the combination of regular chat, connected apps, a sandbox, and a broad message allowance is unusually strong in OpenAI's product.

This is a product-ergonomics assessment as of July 2026, not a permanent model ranking.

A better verification ladder

  1. Unit tests for pure modules.
  2. Backend integration through Node fetch.
  3. Frontend browser tests with a mock API.
  4. A browser-to-Node bridge with the real backend.
  5. One true E2E check in an external environment.

A good assistant should not pretend an unavailable check was completed. It should map the environment, run the maximum available verification, and leave only the genuinely impossible layer to external CI.

Conclusion

Regular ChatGPT Chat is significantly underestimated as a development tool.

It is not the Codex IDE and it does not receive access to the user's Docker installation, GPU, or local terminal. Even so, it can work with a real GitHub repository, write code, run pure backend tests, verify persistence, start local HTTP processes, inspect APIs through Node, execute frontend code in Chromium, assert DOM and CSS behavior, take screenshots, and write the result back to the repository.

For a small or medium task, Codex is often convenient but not mandatory. A substantial part of development can happen in regular Chat, preserving the shared Work and Codex agentic allowance for tasks that genuinely need an autonomous IDE-like environment.

Код без Codex: как далеко можно зайти в обычном ChatGPT

ChatGPT обычно воспринимают как универсального собеседника, а серьёзную работу с репозиторием отправляют в Codex. Но Codex и ChatGPT Work используют общий агентский пул, тогда как обычные сообщения Chat учитываются отдельно. Отсюда простой вопрос: сколько реальной разработки можно выполнить, вообще не включая агентский режим?

Эта статья о проверке внутри обычного ChatGPT Chat, а не Codex, не Work и не локального агента в IDE. Использовались GitHub-коннектор, файловая песочница и доступные инструменты выполнения.

Три режима, которые легко спутать

  • ChatGPT Chat — обычный диалог. Он может использовать подключённые приложения и плагины, работать с файлами, выполнять код в песочнице и совершать разрешённые внешние действия.
  • ChatGPT Work — отдельный режим для длинных исследований и deliverable-задач.
  • Codex — специализированный coding agent в ChatGPT, IDE и CLI.

OpenAI прямо указывает, что Codex, ChatGPT Work и некоторые другие агентские продукты расходуют общий agentic usage pool. Обычные сообщения Chat считаются отдельно; для Plus и Pro сообщения и взаимодействия обозначены как unlimited с оговоркой про abuse guardrails.

Интересная граница проходит не между «чатом» и «кодом», а между ограниченной задачей в Chat и автономной работой в агентской среде.

Почему обычный Chat умеет больше, чем давать советы

Через подключённое GitHub-приложение ChatGPT может читать файлы репозитория, смотреть коммиты и diff, создавать и обновлять файлы, писать изменения в ветку и переносить материалы в песочницу для локального анализа.

Этого уже достаточно для большого класса задач:

  • планы проекта, AGENTS.md и skills;
  • code review и анализ регрессий;
  • точечные исправления и небольшие модули;
  • unit-тесты и анализ test output;
  • README и техническая документация;
  • работа с GitHub без ручного копирования файлов.

Chat не получает доступ к железу пользователя, локальной IDE, Docker daemon и неограниченной сети. Но для большинства небольших и средних задач это не требуется.

Эксперимент: Link Saver в обычном Chat

Тестовым проектом стал небольшой Link Saver: Node.js и Express на backend, обычный JavaScript на frontend, JSON persistence, node:test и GitHub-коннектор.

Цель была исследовательской: получить код, выполнить все доступные проверки и понять, насколько близко обычный Chat может подойти к настоящему browser-to-server E2E.

Первое ограничение: GitHub-коннектор не равен git clone

Коннектор работал с удалённым репозиторием, но не предоставлял одной операции для полной материализации проекта в файловую систему песочницы.

Прямой git clone тоже не сработал: терминал не мог разрешить github.com. Локальную копию пришлось собирать из файлов, прочитанных через отдельный доверенный канал коннектора.

GitHub connector access != network access from the runtime

Интеграция ChatGPT с GitHub и исходящий интернет контейнера — разные поверхности доступа.

Второе ограничение: runtime есть, а сети зависимостей может не быть

В песочнице был Node.js, но не было готового pnpm и node_modules. Corepack присутствовал, однако не мог скачать пакетный менеджер и зависимости из npm registry.

Код на стандартной библиотеке Node.js всё равно запускался:

  • URL-валидация;
  • JSON-хранилище;
  • работа с файлами;
  • создание и удаление записей;
  • проверка persistence после повторной инициализации.

Части с Express и HTML parser из npm без зависимостей не стартовали. Значит, границу задавало содержимое песочницы, а не сам режим Chat.

Третье ограничение: Chromium есть, но localhost может быть закрыт

Chromium и Playwright были доступны. Браузер запускался, выполнял JavaScript, видел DOM, вычислял CSS и делал скриншоты.

Прямой переход на локальные адреса блокировался:

http://127.0.0.1:<port>/
http://localhost:<port>/
http://0.0.0.0:<port>/
file:///.../index.html

Chromium возвращал:

net::ERR_BLOCKED_BY_ADMINISTRATOR

Тот же локальный сервер отвечал через curl. Приложение работало, а блокировка находилась именно на уровне browser-navigation policy.

Самой полезной находкой оказался page.setContent(). HTML можно передать прямо в Chromium, затем подключить реальные CSS и JavaScript. Playwright всё ещё может искать элементы, нажимать кнопки, вводить текст, проверять состояния формы, читать computed styles, собирать console errors и делать скриншоты.

Мост между браузером и реальным backend

Playwright page -> fetch adapter -> Node bridge -> Express API -> JSON storage

Браузер может вызывать переопределённый window.fetch, который через page.exposeFunction() передаёт запрос в Node.js. Уже Node обращается к реальному Express-серверу и возвращает ответ странице.

Это не строгий стопроцентный E2E, потому что HTTP-переход выполняет Node. Но функционально проверяются реальный frontend, реальный backend, persistence, DOM и пользовательские сценарии loading, error, delete и favourites.

Когда обычного Chat достаточно

Chat хорошо подходит для ограниченной работы, которую можно разложить на проверяемые шаги: документация, небольшая функция или исправление, code review, unit- и API-тесты, статический frontend, GitHub-операции и анализ файлов, логов и test output.

Codex разумнее использовать для длинной автономной работы по большому дереву кода, IDE/CLI-workflow, параллельных agents и worktrees, сложных migrations, активной установки зависимостей, локальных developer tools и задач, где постоянное ручное управление дороже agentic лимита.

Work полезен для длинных исследований и deliverable-задач, где агент должен самостоятельно собрать большой итоговый результат.

Режим нужно выбирать по требуемой автономности и инфраструктуре, а не просто по признаку «в задаче есть код».

Почему у OpenAI здесь редкая продуктовая связка

Речь не о том, что модели конкурентов хуже пишут код. Речь о сочетании возможностей продукта.

Один обычный диалог ChatGPT может объединить сильную reasoning-модель, широкий лимит сообщений, подключённые приложения с read/write actions, файловую песочницу, выполнение кода, browser runtime, скриншоты и запись изменений обратно в репозиторий.

Google AI Studio предлагает мощный playground, code execution, Google Search, URL Context, Computer Use и File Search. Но официальная документация описывает AI Studio прежде всего как среду прототипирования Gemini API. Пользовательские интеграции требуют function calling и исполнения в собственном приложении; эквивалентный готовый GitHub-коннектор с write actions не указан среди built-in tools.

Официальное приложение Gemini тоже способно на многое, но его usage limits заметнее влияют на длинные итеративные сессии. В этом сценарии сочетание обычного чата, connectors, sandbox и широкого message limit у OpenAI выглядит особенно сильным.

Это оценка продуктовой эргономики на июль 2026 года, а не вечный рейтинг моделей.

Правильная лестница проверок

  1. Unit-тесты чистых модулей.
  2. Backend integration через Node fetch.
  3. Frontend browser tests с mock API.
  4. Browser-to-Node bridge с реальным backend.
  5. Один настоящий E2E во внешнем окружении.

Хороший помощник не должен притворяться, что выполнил недоступную проверку. Он должен определить границы среды, выполнить максимум доступного и оставить внешнему CI только действительно невозможный слой.

Вывод

Обычный ChatGPT Chat сильно недооценён как инструмент разработки.

Это не Codex IDE, и он не получает доступ к пользовательскому Docker, GPU или локальному терминалу. Но даже при этих ограничениях Chat может работать с настоящим GitHub-репозиторием, писать код, запускать чистые backend-тесты, проверять persistence, поднимать локальные HTTP-процессы, анализировать API через Node, выполнять frontend-код в Chromium, проверять DOM и CSS, делать скриншоты и записывать результат обратно в репозиторий.

Для небольшой или средней задачи Codex часто удобен, но не обязателен. Значительную часть разработки можно выполнить в обычном Chat, сохранив общий лимит Work/Codex для задач, где действительно нужна автономная IDE-подобная среда.
